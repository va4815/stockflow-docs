# ADR-001: Modular Monolith Architecture

| Status | Created | Updated |
|--------|---------|---------|
| Accepted | 22-Jul-2026 | 22-Jul-2026 |

## Context

- StockFlow is an order and inventory management platform
- The system contains multiple business domains, such as authentication, product, inventory and order management
- The architecture should provide clear module boundaries while remaining simple to develop, test and deploy
- The architecture should support future scalability without introducing unnecessary complexity

## Options Considered

| Option | Description |
|--------|-------------|
| `Traditional Monolith` | Single application with limited module separation |
| `Modular Monolith` | Single application organised into independent business modules |
| `Microservices` | Independent services for each business domain |

## Decision

- Adopt a `Modular Monolith Architecture`
- Organise the application into independent business modules
- Deploy the application as a single executable
- Define clear module boundaries and responsibilities
- Allow future migration of modules into microservices if required

## Reason

- Better separation of business domains than a traditional monolith
- Lower operational complexity than microservices
- Easier to develop, test and deploy
- Suitable for the current project size and team
- Provides a clear migration path for future growth

## Consequences

### Positive

- Clear module ownership
- Improved maintainability
- Simplified deployment
- Easier future evolution

### Negative

- Entire application is deployed together
- Modules cannot be scaled independently
- Module boundaries must be enforced through development practices
